# Buddy 游戏化 — 从工具到伙伴的产品思考

## 核心问题：AI 助理缺乏"长期关系感"

大部分 AI 产品的交互是一次性的：用户提问、AI 回答、关闭窗口。即使有记忆，也只是"记住了上次说的话"，缺乏**情感投射**和**持续陪伴的错觉**。

CodePilot 助理想解决的问题是：**怎么让用户愿意每天打开助理，而不只是"需要的时候才想起它"？**

答案是把 AI 包装成一个有生命感的宠物伙伴（Buddy），让功能性行为（记忆、提醒、定时任务）变成"Buddy 在帮你做事"。

## 用户旅程全链路

### 第一阶段：初见（Onboarding）

```
新用户打开 CodePilot
  │
  ├─ 空状态页：双入口卡片
  │   ├─ 📂 项目对话（打开文件夹，AI 帮你写代码）
  │   └─ 🥚 个人助理（设置一个了解你的 AI 助理）
  │       └─ 无 buddy → 卡片上展示 3D 蛋图片
  │
  └─ 用户选择"个人助理"
      │
      ├─ 无工作区 → 设置页选择目录
      └─ 有工作区 → 弹出 Onboarding Wizard（3 步）
          │
          ├─ Step 1：关于你（名字 + 角色选择）
          ├─ Step 2：关于助理（名字 + 沟通风格 + 边界）
          └─ Step 3：Buddy 揭晓
              ├─ 3D 物种图片（Fluent UI emoji via CDN）
              ├─ 稀有度渐变背景 + 胶囊标签
              ├─ 五项属性条（peak stat 高亮）
              └─ "开始聊天 →"
```

**设计思考**：

- Wizard 零 API 调用，全前端完成，秒级体验。用户不需要等待 AI 响应就能看到 Buddy
- Buddy 揭晓放在最后一步，作为"奖励"——完成了自我介绍就能见到伙伴
- 稀有度机制创造"开箱"惊喜：1% 传说概率让用户有分享欲
- 16 个物种覆盖不同审美偏好，但不要太多（避免选择困难）

### 第二阶段：初次对话

```
Wizard 完成 → 自动创建助理会话 → 跳转
  │
  ├─ 会话为空 + 有 Buddy → 自动触发 buddy 首次自我介绍
  │   AI 用 buddy 的物种/性格/属性做开场白
  │   例："嗨！我是一只好奇的猫咪 🐱 最擅长帮你发现问题~"
  │
  └─ 会话为空 + 无 Buddy → 自动触发领养引导
      展示蛋 Widget → 等待用户说"孵化" → 生成 Buddy → 起名
```

**设计思考**：

- 空会话不应沉默——这是建立第一印象的时刻
- 两种空状态互斥：有 Buddy 走自我介绍，无 Buddy 走领养引导
- 自我介绍不是硬编码文本，而是 AI 根据 buddy 属性自由发挥，每次不同
- 领养引导刻意设计成"对话式"而非"表单式"，让用户感觉在和蛋互动

### 第三阶段：日常使用

```
用户打开助理会话
  │
  ├─ 系统 prompt 注入：
  │   ├─ 身份层（soul.md + user.md + claude.md）
  │   ├─ Buddy 性格 prompt（根据物种/属性/稀有度生成）
  │   ├─ 心跳检查提示（如果今天还没做）
  │   └─ 渐进式文件更新指引
  │
  ├─ AI 消息左侧显示 Buddy 头像
  ├─ 记忆 MCP 工具随时可用（搜索/读取/最近记忆）
  └─ 定时任务/通知 MCP 工具随时可用
```

**心跳系统的两种模式**：

1. **完整心跳（autoTrigger）**：空会话 + 心跳过期 + 有 Buddy → 自动发送不可见消息，AI 检查 HEARTBEAT.md 并自主决定说什么
2. **软心跳（系统 prompt hint）**：用户主动发消息 + 心跳过期 → prompt 中附加轻量提示，AI 自然地在回答中顺带检查，完成后输出 `<!-- heartbeat-done -->` 不可见标记

**设计思考**：

- 心跳不应打断用户——优先回答用户问题，顺带做检查
- 完整心跳只在空会话触发，不会在对话中途弹出
- `<!-- heartbeat-done -->` 标记在持久化前被清除，用户和后续上下文都看不到
- Buddy 不存在时心跳不触发，避免和领养引导冲突

### 第四阶段：成长与进化

```
随时间推移：
  │
  ├─ 记忆积累 → 自动提取（每 3 轮，Epic+ 每 2 轮）
  ├─ 对话次数增加
  ├─ 活跃天数增加
  │
  └─ 达到进化条件 → 看板"检查进化"按钮
      ├─ 条件满足 → 稀有度提升 + 属性增长 + 庆祝
      └─ 条件不满足 → Toast 提示差多少（记忆 X/Y, 天数 X/Y）
```

**进化路径**：

| 路径 | 记忆 | 天数 | 对话 |
|------|------|------|------|
| Common → Uncommon | 10 | 7 | 20 |
| Uncommon → Rare | 30 | 21 | 50 |
| Rare → Epic | 60 | 45 | 100 |
| Epic → Legendary | 100 | 90 | 200 |

**设计思考**：

- 进化不是自动的，需要用户主动"检查"——这本身就是一种参与仪式
- 门槛设计参考了 Tamagotchi 和 Pokémon 的成长曲线：早期容易、后期需要真正的日常使用
- 进化不只是视觉变化，还解锁实际能力（增强性格、记忆加速）
- 失败反馈告诉具体差多少，给用户"再坚持几天"的动力

### 第五阶段：通知与任务

```
用户或 AI 创建定时任务
  │
  ├─ 到期 → 调度器执行 → AI 生成结果
  │   ├─ 结果插入聊天（带 Buddy 前缀）
  │   └─ 通知推送：
  │       ├─ low → 应用内 Toast
  │       ├─ normal → Toast + Electron 系统通知
  │       └─ urgent → Toast + 系统通知 + Telegram
  │
  └─ 通知流：
      服务端队列 → 前端 5 秒轮询 → Toast + 系统通知
      ├─ Electron 环境 → electronAPI.notification.show（支持点击回到窗口）
      └─ 浏览器环境 → Web Notification API（dev 模式 fallback）
```

**设计思考**：

- 通知消息带 Buddy 前缀，让用户感觉是"伙伴在提醒你"而非"系统在推送"
- 三级优先级覆盖不同场景：喝水提醒(low) vs 会议提醒(normal) vs 紧急事项(urgent)
- 服务端直接调用 `sendNotification` 而非 HTTP 绕行，避免端口问题
- 前端 5 秒轮询是简单可靠的方案，未来可以升级为 SSE

## 视觉体系

### 3D Fluent Emoji

所有 Buddy 相关的视觉都使用微软 Fluent UI 3D emoji（via jsdelivr CDN），而非系统 emoji：

- **为什么**：系统 emoji 跨平台不一致（Apple/Google/Windows 差异大），3D emoji 有统一的高品质视觉，更适合"游戏角色"感
- **哪里用到**：Wizard 揭晓、聊天空状态、新建聊天入口、侧栏助理卡、看板 Buddy 卡、侧栏推广卡
- **蛋图片**：无 Buddy 时统一用 3D 蛋图片，暗示"等待孵化"

### 稀有度渐变系统

每个稀有度有一套完整的渐变色：

| 稀有度 | 渐变色 | 视觉感受 |
|--------|--------|---------|
| Common | 灰色 → 浅灰 | 朴素 |
| Uncommon | 绿色 → 翠绿 | 清新 |
| Rare | 蓝色 → 天蓝 | 冷静 |
| Epic | 紫色 → 薰衣草 | 神秘 |
| Legendary | 金色 → 暖金 | 华贵 |

渐变色用于：
- Wizard Step 3 揭晓背景
- 看板 Buddy 卡边框
- 稀有度胶囊标签背景
- 聊天空状态 Buddy 背景

## 这轮审查修复的关键决策

### 心跳系统重构

**原问题**：心跳指令混入普通对话，前后端判定不一致，buddy-welcome 和 heartbeat 冲突

**最终设计**：
- 前端 `useAssistantTrigger` 只在空会话触发，buddy-welcome 和 heartbeat 通过 `!!state.buddy` 互斥
- 服务端 `needsHeartbeat` 字段是唯一数据源（包含 buddy 存在性检查），前端不再重新实现判定逻辑
- `context-assembler` 通过 `userPrompt.includes('心跳检查')` 区分完整心跳和 buddy-welcome，不再仅依赖 `autoTrigger` flag
- 软心跳用 `<!-- heartbeat-done -->` 不可见标记闭环，持久化前清除

### 安全修复

- `codepilot_memory_get` 增加 `fs.realpathSync` 解析 symlink 后再验证路径，防止 symlink 逃逸读取工作区外文件
- 自动记忆提取的"本轮已写入"检查改为传入完整 `contentBlocks` JSON（含 tool_use 块），而非纯文本

### 调度器健壮性

- 冷启动恢复：`instrumentation.ts` 中启动调度器，不再依赖 `/api/chat` 首次访问
- Session-only 任务：循环任务推进 `next_run`、失败退避/自动禁用、可列出/可取消
- Cron 表达式：统一 4 年日级扫描（预检 dom/month/dow 跳过不匹配日），无匹配返回 `null` → 任务暂停，而非伪造执行时间
- 硬编码端口：所有 MCP 工具改用 `getBaseUrl()`（读 `process.env.PORT`），通知和调度改为直接 import 避免 HTTP 绕行

### 通知链路

- 服务端通知队列（globalThis ring buffer）+ `GET /api/tasks/notify` 轮询 + `useNotificationPoll` 前端 hook
- Electron 环境优先走 IPC 原生通知（支持点击回到窗口），浏览器 Notification 仅作 dev 模式 fallback
- 类型定义 `electron.d.ts` 补充了 `notification` 接口

## 与 Claude Code 宠物系统的对比

Claude Code（CLI 版本）在 2025 年末引入了类似的宠物机制：用户在终端中会遇到一个像素风虚拟宠物，随使用时间成长。这验证了"给 AI 工具加宠物"这个方向的可行性，但 CodePilot 作为**桌面 GUI 客户端**，能做到很多 CLI 做不到的事。

### 形态差异

| 维度 | Claude Code（CLI） | CodePilot（桌面客户端） |
|------|-------------------|----------------------|
| 视觉 | 终端 ASCII / 像素风，受限于字符渲染 | 3D Fluent Emoji + 渐变色背景 + CSS 动效，完整的视觉表现力 |
| 交互 | 命令行文本交互，宠物只能"出现在输出里" | 独立 UI 区域：看板卡片、侧栏头像、聊天气泡、系统通知 |
| 持久可见性 | 只在命令执行时短暂出现 | 始终可见：侧栏展示名字、看板展示状态、顶栏展示 emoji |
| 通知能力 | 无（终端关了就没了） | 三级通知：Toast + Electron 系统通知 + Telegram |
| 状态面板 | 无法展示复杂 UI | 完整看板卡：属性条、进化进度、心跳/记忆/任务状态 |

### 功能深度差异

**记忆系统**：Claude Code 的宠物没有独立的记忆架构。CodePilot 的 Buddy 绑定了一套完整的持久化记忆系统——daily memory、长期记忆、自动提取、Obsidian 语法感知。Buddy 的稀有度直接影响记忆提取频率（Epic+ 每 2 轮 vs 普通每 3 轮），让"养宠物"和"用工具"产生真实关联。

**心跳/主动性**：CLI 宠物是被动的（用户不执行命令就不出现）。CodePilot 的 Buddy 有自主心跳——每天第一次打开会主动检查待办事项、回顾记忆、提醒跟进，像一个真的在等你的伙伴。软心跳模式甚至能在普通对话中"顺带"完成日常检查，不打断用户。

**定时任务**：CLI 没有后台调度能力（进程退出即停止）。CodePilot 的 Buddy 可以创建持久化定时任务（cron/interval/once），服务端 10 秒轮询执行，跨重启保留。"🐱 Toki 提醒你：该喝水了~" 这种体验在 CLI 里做不到。

**进化系统**：CLI 的成长通常是线性的（使用时间 → 等级）。CodePilot 的进化有三个维度（记忆数 × 活跃天数 × 对话数），并且进化不只是数字变化——稀有度提升会解锁增强性格、记忆加速等实际能力，soul.md 中的性格 prompt 也会随之更新。

### 桌面客户端独有的优势

1. **Electron 原生通知**：Buddy 的提醒可以弹系统通知栏，点击直接回到应用窗口。CLI 做不到这种"后台运行、前台提醒"的体验
2. **Generative UI Widget**：孵化/进化/里程碑可以用 show-widget 渲染丰富的 HTML 卡片（动画、渐变、交互按钮），CLI 只能输出文本
3. **多面板布局**：Buddy 状态卡和聊天共存于同一屏，用户可以一边对话一边看 Buddy 的状态。CLI 是串行输出，看完就没了
4. **持久化看板**：Buddy 的属性、进化进度、任务列表始终可见可交互，不需要执行命令才能查看
5. **IPC 桥接**：Electron main process 可以在后台运行调度器、发送系统通知、管理文件 watch，这些都是纯 CLI 进程做不到的

### 本质区别

Claude Code 的宠物是一个**彩蛋**——在已有 CLI 工具上增加的趣味元素，核心价值还是 CLI 本身。

CodePilot 的 Buddy 是一个**设计支点**——整个助理产品围绕"有生命感的伙伴"来设计，记忆、心跳、定时任务、通知都是 Buddy "做的事情"。不是给工具贴了个宠物皮肤，而是让宠物成为工具的使用界面。

这也是为什么我们选择桌面客户端形态：**一个有生命感的伙伴需要一个有空间的家**。终端太小、太临时、太串行，放不下一个值得每天回来看一眼的伙伴。

## 已知局限与未来方向

### 当前局限

1. **孵化/进化动画缺失**：目前是静态揭晓，没有蛋裂开、闪光等过渡效果
2. **Buddy 首次自我介绍未实现**：Wizard 完成后进入聊天是空白（只有 Quick Actions），缺少"首次见面"的仪式感
3. **通知轮询不够实时**：5 秒间隔对紧急通知有延迟，可考虑升级为 SSE
4. **进化只有数值变化**：缺少视觉动效（形态过渡、闪光、粒子）
5. **里程碑消息是纯文本**：没有用 show-widget 做游戏化卡片
6. **Buddy 重置后 soul.md 清理依赖正则**：`## Buddy Trait` 节的移除用字符串匹配，复杂格式可能残留

### 未来方向

1. **孵化仪式**：蛋摇晃 → 裂开 → 揭晓的 CSS 动画流程
2. **Buddy 起名环节**：孵化后在 Widget 中内嵌名字输入，而非通过对话
3. **进化动画 Widget**：旧形态 → 光效 → 新形态的 show-widget
4. **通知气泡样式**：任务结果用 Buddy 说话气泡展示，而非纯文本前缀
5. **Buddy 互动**：摸头、喂食等小互动增加情感连接
6. **多 Buddy 系统**：收集不同物种，切换当前 Buddy
7. **SSE 通知推送**：替代轮询，实现即时通知
